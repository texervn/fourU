fourU by xyztextbooks

A rewrite of the WeBWorK online homework system, in Django

About

fourU is a online homework system, giving teachers the ability to assign problems with randomly-generated numbers, and students the ability to get immediate feedback on answers.

Goals

fourU was born out of frustration with the current contenders in the online homework system arena; xxx and yyy were too expensive for a small company, and WeBWorK provided constant attention from James, whose official job title was "Web Programmer", but should have been "Web Programmer / System Administrator / Technical Support / General Technology Guy". It also provided a less-than-friendly interface, both for professors and students.

These, then, are primary goals of the fourU project:

  • As few dependencies as possible. Required package versions should be either available in the repositories of the latest versions of Debian Stable and CentOS, or be easily installable by a non-sys-admin on common systems.
  • Support for Windows, Mac OS X and Linux. Attempts should be made to insure that fourU works on other operating systems, such as FreeBSD and Solaris, if they don't require extraordinary amounts of time or are otherwise detrimental to the project.
  • Support for a wide variety of web servers. One of WeBWorK's downfalls is a reliance on Apache; Even Apache 2.2 support is experimental! Most installations don't need the advanced features that Apache provides, and so we encourage our users to try out one of the lightweight alternatives.
  • Separation of backend logic and frontend presentation. It should be easy for a non-programming designer to customize the appearance of your site, with less than an hour invested in learning the templating system.
  • Clear, modular, well-organized, fully-documented code. It's hard enough convincing people to work on your code base as it is; we want to lower the entry barrier as much as possible.
  • Unit tests that cover a large part of the code base. Automated regression testing on a multitude of different platforms and configurations is a great way to prevent "This worked in version x.xx, but now..." messages.
  • Up-to-date documentation. It's always frustrating when you're setting up a new piece of software, and the instructions you're reading don't match what you see. Programmers notoriously hate writing documentation, but we won't merge in patches to the code base without patches to the docs, if they will introduce any changes visible to the end-user.
  • A "Worse is Better" approach to UI design. Influenced much by Steve Krug's "Don't Make Me Think", we want to provide the user with only a few, simple choices at any given time. If you need to drill-down some specifics for one of those, you can ask them later.

Questions

Q: Why write this yourself, rather than use the existing WeBWorK codebase? Joel says you should never do this!

A: Of the three reasons Joel mentions programmers give for rewriting code, the "architectural problems" are the closest to what we felt with WeBWorK.

  1. Much of the code would not actually be rewritten, but rather use the already rewritten (and proven) implementations present in the framework and various libraries.
  2. James doesn't know Perl nearly well enough to undertake something of that magnitude :). A short experiment was tried, and we found that a) we couldn't find any local Perl hackers and b) outsourced Perl programmers at the price we needed required far too much micromanagement. That, then, was the end of Perl.
You'll note that rewrites are sometimes extremely successful: both the Windows NT kernel (upon which XP was based) and Mac OS X are widely accepted as being tremendous improvements over their predeccesors.

Q: Ok, but why Python?

A: Well, that all depends on what you're thinking it should have been instead. ;)

Java and .NET have far too much overhead to be either developed or deployed quickly and easily. Rails has a tendency to be both magical and opinionated, both things that James feels make development difficult (although, obviously, many people have done very well with it). More importantly, Ruby's userland libraries are still in a fairly young stage; they're growing at an incredible rate, but they couldn't compare (at the time the decision was made) to the wealth of Python libraries.

PHP was the initial decision, actually. It's a dynamically-typed language with a large number of libraries, code snippets and users, and was designed for the web specifically. It's also one of the most common server-side languages, and is available on free and cheap shared hosts everywhere. After having forsaken PHP for a year for the Python creed, however, James found himself constantly doing all in his power to use a PHP version of Django, and so, he decided to use the real thing instead.

Q: What about [insert Python framework here]?

A: Django has:

  • A 1.0 release, with guarenteed API stability
  • A large userbase
  • Active development
  • Excellent and extensive documentation
  • A wide variety of 3rd-party middlewares
  • Tightly-coherent components that allow you to get going with minimal fuss
  • ...and loosely-couple components so that you can swap out pieces as needed.
